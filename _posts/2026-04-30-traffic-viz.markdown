I built a browser-based 3D visualization tool for SUMO traffic simulations that turns raw trajectory data into interactive, cinematic scenes—without requiring a game engine. I will briefly walk you through the app and highlight some features. Source code can be found at the [project page on GitHub](https://github.com/jeroenvanriel/traffic-viz), where you may also find precompiled binaries.

## Showcasing research on autonomous intersections

This project started with a simple request from my thesis supervisor: "Can we make a nice 3D traffic visualization tool to showcase our research on autonomous intersections?"

He is interested in mathematical models for road traffic. At the time, his [latest research](https://www.tandfonline.com/doi/full/10.1080/23249935.2019.1692962) ventured into platoon forming algorithms for autonomous intersections, in which vehicle trajectories are fully planned in advance rather than being a consequence of reactive human driving behavior.
In less abstract terms, we want to retire the traditional traffic light and instead have vehicles coordinate with each other to pass through the intersection safely, which can be shown to lead to significant improvements in traffic flow.

While it is certainly possible to visualize such systems in 2D or through basic trajectory plots, we want something more intuitive and visually engaging. A nice 3D scene where vehicles move through a realistic network of intersections is just visually much more pleasing and can be more effective in communicating the underlying principles to a wider audience, including non-experts.

Since we were already using the [SUMO traffic simulator][sumo] in our research, it made sense to build on its existing file formats for road networks and vehicle trajectories. That immediately gave us interoperability with standard traffic modeling tools, while still leaving freedom in how the 3D scene is rendered. From this starting point, I began developing a visualization tool that reads SUMO network and trajectory data and renders it as an interactive 3D scene in the browser.

## Feature walkthrough

The first thing you see when you open the app in a browser is a list of available scenes. Each scene is defined by a pair of a *network file*, specifying the geometry of the road network, and a *trajectory file*, which encodes a set of vehicle trajectories. A little preview of the network layout is shown for each scene in the overview.

After selecting a particular scene, you get into the main *scene view*, where the 3D visualization happens. You can move the camera with map-like controls and view the vehicle animation using the timeline slider and some additional controls. Because trajectories are often sampled at coarse time steps, naive playback would look jerky. That is why I implemented interpolation to reconstruct smooth motion in real time.

### Generating and editing scenes

Relying on SUMO's [network file format][sumo-network-file] enables us to depend on existing tooling for generating, viewing and editing network files. The command-line tool [netgenerate][netgenerate] can be used to generate parameterized networks (grid, spider, etc.), with optional randomization. There is also a Python script ([osmWebWizard.py][osmwebwizard]) to import parts of real-world networks from OpenStreetMap. Using the [sumolib][sumolib] Python package and SUMO's built-in [netconvert][netconvert] tool, it is also relatively easy to write your own network generator. Finally, SUMO's built-in [netedit][netedit] program allows you to manually create and edit network files graphically.

As data format for the vehicle trajectories, we chose to use SUMO's [floating car data][sumo-fcd] format, which basically lists the vehicle positions for each time step of the simulation. This file can be directly generated by SUMO for a given simulation sceneario. Instead of using SUMO, with its reactive car-following model, the original motivation was to be able to generate the trajectory file manually. Since the floating car data format is just simple XML, this is a routine task in any common programming language.

### Camera paths and video recording

In order to produce nice cinematic videos, I added a system for automating the camera during playback.
To define a custom camera path, you can switch to the *camera editing* mode, where you can add keyframes at specific timestamps and specify the camera position and orientation for each keyframe. During playback, the camera will then automatically move along the defined path, creating a smooth animation that follows the action in the scene.
You can choose between basic linear interpolation or Catmull-Rom spline interpolation for a smoother path.

To avoid having to record the screen with external tools, I also implemented a simple video recording feature. When you start recording, the app will play through the animation while capturing the frames. When you stop recording, it will compile the captured frames into a video that you can download.
It will produce a `.webm` video file, but this can be easily converted into other formats like `.mp4`, for example using FFmpeg.
Currently, this feature only works in Chrome; I have had problems getting this to work with other browsers so far.

### Vehicle model customization

It is possible to use your own 3D vehicle models by uploading them to the *model library*. For example, you might want to use a different model for different vehicle types, such as passenger cars, taxis, trucks, busses, etc. To use custom models, you need to specify the *type* property in the trajectory file, which is a string that can be used to differentiate between different vehicle types. The app will then look for a model with the same name in the model library and use it for rendering that vehicle. If no matching model is found, a default box-shaped model will be used.

Since most 3D models found online do not already have the proper dimensions and orientation, you can use the *model configurator* to adjust the reference point, scale and rotation of a custom vehicle model. This is especially crucial to achieve a realistic animation of turning around corners. When you make changes in the configurator, they are immediately applied in all running animations, which helps with manual tweaking.

### Some nice visual details

To prevent the awkward sight of moving vehicles with static wheels, I invested some time to get spinning wheels.
To use this feature, the custom model's *internal scene* should contain separate wheel objects. Since most models do not have this, it might be necessary to manually edit the model file (e.g., using Blender) to split the wheels into separate objects.
The wheel radius is then automatically estimated and used to calculate the current rotation speed to create a realistic animation.
At the time of writing, turning wheels are not yet supported, since the original use case only involved intersections of straight roads, but this is something I would like to add in the future.

Lastly, I also added a straightforward *fake shadow* to vehicles. Real shadows can become quite expensive in terms of graphical processing, so a static shadow gradient is drawn below the moving object. The size of the shadow is automatically estimated based on the models footprint.

## Project background

This project started somewhere back in 2021, when I was finishing my Bachelor's thesis. My thesis supervisor at the time asked me if I could develop a simple 3D visualization tool to promote their research on platoon forming algorithms. Part of this project has been funded by the [Dutch Research Council (NWO)](https://www.nwo.nl/en) through the ["Implementing Stochastic Models for Intersections with Regulated Traffic" (ISMIRT)](https://www.nwo.nl/en/projects/43920616) project and I presented a preliminary version of the app at the ["ISMIRT Workshop on Urban Traffic Management and Control"](https://www.utwente.nl/en/eemcs/sor/events/symposia/220610_ISMIRT%20workshop%20on%20Urban%20Traffic%20Management%20and%20control/).

### Extending an existing project

To avoid reinventing the wheel, I started looking at some existing 3D projects for SUMO. Some of them are based on powerful game engines like Unreal and Unity or use the [CARLA simulator][carla] for very realistic graphics. Since we were mostly after lightweight tooling with minimal setup and 3D modeling, we decided to extend the [sumo-web3d](https://github.com/sidewalklabs/sumo-web3d) project, which was develped at Sidewalks Labs, a former urban innovation subsidiary of Google.

The sumo-web3d project fit our purposes because it is lightweight and it was already based on SUMO's road network file format. However, it's main purpose was to visualize a live running SUMO simulation, whereas we were interested in using vehicles trajectories that are generated ahead of time, so I added support for SUMO's floating car data file format for offline simulations.

I spend some time improving the appearance of the road by adding edge lines and dashed lane separation lines. Furthermore, I added features to draw vehicle colors depending on their speed, and to show a kind of "lane pressure" as a gradient color. The latter was specifically requested to visualize the underlying queuing models described in [this PhD thesis](https://research.utwente.nl/en/publications/queueing-models-for-urban-traffic-networks/). It makes me very proud that this version of the tool has also been used for [this article](https://dl.acm.org/doi/full/10.1145/3701042) (see the supplementary material) on dealing with mixed traffic in autonomous intersections.

### Time for a full rewrite, with LLM assistance

Some time ago, I decided it was time to start rewriting the app from scratch. The main reasons were that sumo-web3d contained a lot of features that we did not need, it depended on some old unmaintained packages that would require rather painful migration, and I thought the redesign itself would be a great learning experience for me.

Initially, the choice for a web application has been mainly motivated by the existing sumo-web3d project, but it still seemed a logical choice at the time I started the rewrite.
I used the FastAPI library in a Python backend that exposes some simple data endpoints, caching mechanisms and some procedures for parsing network and trajectory files, which are computationally somewhat more demanding.
The frontend is built using the React framework and uses the three.js library for 3D rendering. 

This rewrite was my first project for which I heavily used LLM technology, namely ChatGPT for ideation and brainstorming and GitHub Copilot for coding. I have been quite impressed by the speed at which I have been able to reimplement features, but I also like to think that this was at least in some part due to the fact that I already had a clear idea about what the overall application should look like, which certainly helped to write good prompts. I think that the huge popularity of React for frontend development is also one of the main reasons why LLM-assisted coding works so well there.

### Next features

The new app already serves most of our original needs. However, there are still some features that would be nice to have, which I will briefly list here.

- *Procedural scenery*. A lot of the related projects listed above allow you to create a very realistic environment, including buildings, vegetation, traffic signs and lights. Although our goal is not to aim at maximally realistic scenes, it would be nice to be able to add some background elements to the scene. Because the emphasis is on clearly showing the joint movement of vehicles, I am thinking about drawing some (transparent) building silhouettes that are procedurally generated based on the road network.

- *User-defined scenery*. The earlier version of the app had a checkbox for randomly placed trees, but they were a bit awkard and distracting, probably mainly because the 3d model itself was not that nice. To enable the user to judge and experiment for themselves, a general-purpose mechanism to place any 3d model in the scene would be handy. In other projects, SUMO's [*additional file* format][sumo-additional] has been used for this.

- *Customizable textures*. When working on the previous version of the app, I invested some time in trying to generate a realistic asphalt texture using a custom shader based on Perlin noise. I would like to bring this to the new version with some more customization options, such that the user can tweak the appearence to be realistic or cartoonish to their own liking.

- *Vehicle-following camera*. In some similar projects, it is possible to follow one of the vehicles in a first-person or third-person fashion, so either from within the car or following it from above.

### Related projects

If you are looking for more or different features, know that there are already quite some projects that provide similar base functionality of producing an 3D animation of SUMO networks. Here is a small selection that I think is worth checking out:

- [sumo-web3d](https://github.com/sidewalklabs/sumo-web3d): Interactive web app for visualizing live SUMO simulations.
- [sumo-gui](https://sumo.dlr.de/docs/sumo-gui.html#3d_visualization): Some experimental support for 3D rendering in SUMO itself.
- [SumoNetVis](https://github.com/patmalcolm91/SumoNetVis): Animate 2D vehicle trajectories, create 3D road network (no animation).
- [SUMO2Unity](https://github.com/SimuTraffX-Lab/SUMO2Unity): Visualize live SUMO simulation in the Unity game engine.
- [SUMO2Unreal](https://github.com/AugmentedDesignLab/Sumo2Unreal): Convert SUMO's .net.xml road network file to Unreal game engine.
- [SUMO3d](https://github.com/SvenMertin/SUMO3d): Visualize live SUMO simulation in the Unity game engine.
- [sumo3Dviz](https://github.com/DerKevinRiehl/sumo3Dviz): Visualize pre-recorded SUMO simulations.

## Conclusion

This project shows how lightweight, browser-based tools can be used to turn raw simulation data into intuitive and visually engaging 3D experiences, building on top of the SUMO ecosystem.

More broadly, this project reflects my interest in using visual tools to make complex systems easier to understand. Whether it’s traffic models or other types of simulations, I find it exciting to translate abstract data into something that can be explored intuitively and communicated to a wider audience.

I’m interested in continuing to develop this further, particularly by adding richer environments and more advanced interaction features, and in exploring how tools like this can help make research more accessible and engaging.
If you’re working on similar problems or have feedback, I’d be happy to hear from you.
